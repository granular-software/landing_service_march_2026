What if you make it impossible by design?

Granular agents are constrained by three structural layers.

01
Scope

The agent sees exactly what the user can do. No more.

On each request, the agent only sees what that user is allowed to access. It does not browse a global system. If a resource is outside that user's scope, the agent cannot see it or act on it.

A sales rep's agent sees their accounts, their pipeline, their contacts.
It cannot touch another rep's data, because that data is never in scope.
02
Rules

Even if it tries, the system refuses.

Rules live in the runtime, not in prompt text. When the agent calls an action that violates a rule, execution stops and returns an error. No \"best effort\" behavior, no silent fallback.

Agent: can_refund({ amount: 800, customer: "cust_99" }) // Error: Requires approval - execution stopped
03
Guardrails

Set hard limits for edge cases and abuse paths.

Use system-level caps like volume thresholds, rate limits, and anomaly triggers. When a guardrail is hit, execution stops and your team is notified immediately.

Example: a conversation fetches more than 1,000 objects in a single session. // Guardrail triggered - execution blocked, dev notified
